# Технологический стек

## Назначение документа
Документ фиксирует базовые решения по технологическому стеку и архитектурные принципы CRM. Он служит отправной точкой для
команд разработки, эксплуатации и бизнес-заказчика, помогая синхронизировать ожидания относительно будущих сервисов и
инфраструктуры. Материал дополняет обзорные документы (`README.md`, `docs/architecture.md`, `docs/security-and-access.md`)
и устанавливает общие правила для детальных спецификаций отдельных сервисов.

## Общие архитектурные принципы
- **Микросервисный подход.** Функциональные домены выделяются в самостоятельные сервисы с чёткими контрактами, управляемыми
  через API Gateway/BFF.
- **Единая база данных.** На стартовом этапе все сервисы используют одну реляционную БД с разграничением по схемам и правам
  доступа; в дальнейшем допускается декомпозиция при появлении обоснованных требований.
- **Асинхронные коммуникации.** Для фоновых обработок и интеграций применяются очереди сообщений (RabbitMQ/Kafka) и
  публикация событий, чтобы развязать сервисы и обеспечить устойчивость при пиковых нагрузках.
- **Кеширование.** Чтение справочной и агрегированной информации ускоряется с помощью Redis или аналогичного in-memory кеша
  с контролем согласованности и TTL.
- **Наблюдаемость и управление надёжностью.** Все сервисы обязаны вести структурированные логи, метрики и трассировки
  (например, через Prometheus + Grafana, OpenTelemetry), поддерживать алерты по ключевым показателям и иметь здравый подход
  к обработке ошибок и повторным попыткам.
- **Управление конфигурацией и секретами.** Конфигурация хранится в централизованном хранилище (Vault/SSM) с версионированием,
  секреты не попадают в репозиторий. Развёртывание автоматизируется через CI/CD с обязательными проверками качества.
- **Безопасность по умолчанию.** Реализуются единые механизмы аутентификации/авторизации, шифрование трафика, контроль
  доступа к данным и регулярные проверки на соответствие требованиям безопасности.

## Дальнейшее развитие
Список технологий и конкретные решения будут расширяться по мере детализации сервисов и инфраструктуры. Обновления будут
включать выбор конкретных СУБД, брокеров сообщений, инструментов развёртывания и другие компоненты.
